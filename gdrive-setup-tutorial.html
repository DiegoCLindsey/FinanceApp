Guía completa para activar el backup cifrado en la nube sin exponer credenciales en el repositorio. Actualizada para 2026.

1Google Cloud 2Credenciales 3Repositorio 4Secrets 5Deploy 6Verificar
ℹ️
¿Por qué este proceso? Google Drive requiere credenciales OAuth 2.0 para acceder a la API. Estas credenciales no pueden estar hardcodeadas en el repositorio público (cualquiera podría verlas). La solución es guardarlas como Repository Secrets en GitHub y usar GitHub Actions para inyectarlas en el código justo antes del despliegue, de modo que solo el binario desplegado las contiene, nunca el código fuente.
⚠️
Tipo de cliente OAuth: Desktop app En 2026, Google requiere client_secret para todos los tipos de cliente OAuth, incluyendo SPAs. Para aplicaciones que no pueden guardar secrets de forma segura (como una página estática), Google recomienda el tipo "Desktop app", que trata explícitamente el client_secret como no confidencial. Ver documentación oficial.

1
Crear proyecto en Google Cloud Console
✅
Si ya tienes un proyecto de Google Cloud para esta app, ve directamente al paso 2.
  1. Ve a console.cloud.google.com e inicia sesión con tu cuenta de Google.
  2. En la barra superior, haz clic en el selector de proyectos → "Nuevo proyecto".
  3. Dale un nombre (ej: financeapp-drive) y haz clic en "Crear". Espera a que se cree (10-20 segundos).
  4. Asegúrate de que el proyecto nuevo está seleccionado en el selector de la barra superior.
2
Activar la API de Google Drive y crear credenciales OAuth

Este es el paso más largo. Son cuatro sub-pasos: habilitar la API, configurar la pantalla de consentimiento, crear las credenciales y apuntar los valores.

2a — Habilitar Google Drive API

  1. En el menú lateral izquierdo: APIs y servicios → Biblioteca.
  2. Busca Google Drive API y haz clic en el resultado.
  3. Haz clic en el botón azul "Habilitar".

2b — Pantalla de consentimiento OAuth

  1. Ve a APIs y servicios → Pantalla de consentimiento de OAuth.
  2. Tipo de usuario: Externo → "Crear".
  3. Rellena los campos obligatorios:
    • Nombre de la aplicación: FinanceApp (o el nombre que prefieras)
    • Email de soporte de usuario: tu email
    • Email del desarrollador: tu email
  4. Haz clic en "Guardar y continuar".
  5. En la pantalla "Permisos", haz clic en "Añadir o quitar permisos".
  6. Busca drive.appdata, marca la casilla y haz clic en "Actualizar".
    ℹ️
    ¿Qué es drive.appdata? Es un scope restringido que solo permite a la app acceder a su propia carpeta oculta en el Drive del usuario. El usuario no puede ver estos archivos en drive.google.com y ninguna otra app tampoco puede.
  7. Haz clic en "Guardar y continuar" hasta llegar a "Usuarios de prueba".
  8. Añade tu propio email como usuario de prueba (necesario mientras la app esté en modo "Testing").
  9. Haz clic en "Guardar y continuar" → "Volver al panel".
⚠️
¿Publicar la app? Si es para uso personal, no hace falta publicarla. En modo "Testing" funciona perfectamente para los usuarios de prueba añadidos. Si quieres compartirla con otras personas, deberás publicarla (posible verificación de Google para scopes sensibles como drive.appdata).

2c — Crear credenciales OAuth tipo "Desktop app"

🔑
Usa el tipo "Aplicación de escritorio" (Desktop app), NO "Aplicación web". En 2026, Google requiere client_secret para todos los tipos de cliente. El tipo "Desktop app" es el único que Google reconoce explícitamente como cliente público, lo que significa que el client_secret puede estar en código no confidencial sin implicaciones de seguridad adicionales. Una "Aplicación web" requeriría que el secret fuera confidencial, lo cual es imposible en una SPA estática.
  1. Ve a APIs y servicios → Credenciales.
  2. Haz clic en "+ Crear credenciales" → "ID de cliente de OAuth".
  3. Tipo de aplicación: "Aplicación de escritorio" (Desktop app).
  4. Nombre: financeapp-desktop (o cualquier nombre).
  5. Haz clic en "Crear".
  6. Aparece un diálogo con ID de cliente y Secreto de cliente. Cópialos y guárdalos — los necesitarás en el paso 4.
    🚫
    NO los pongas directamente en ningún archivo del repositorio. Solo en los Repository Secrets de GitHub (paso 4).

2d — Añadir el origen de la página como URI de redireccionamiento

ℹ️
Para tipo "Desktop app", Google acepta http://localhost y puertos como URIs de redirección. También acepta la URL completa de una página web si se registra. Para GitHub Pages debes añadir tu URL.
  1. En la lista de credenciales, haz clic en el nombre de la que acabas de crear.
  2. En "URIs de redireccionamiento autorizados", haz clic en "+ Añadir URI".
  3. Añade la URL exacta de tu GitHub Pages: 
    https://TU_USUARIO.github.io/TU_REPOSITORIO
    Por ejemplo: https://diegoclindsey.github.io/financeapp
  4. Si también quieres desarrollar en local, añade también: http://localhost:8080
  5. Haz clic en "Guardar".
3
Preparar el repositorio en GitHub

El repositorio debe contener la carpeta financeapp/ con el archivo .github/workflows/deploy.yml incluido en el ZIP que descargaste.

  1. Crea un repositorio nuevo en GitHub (puede ser público o privado).
  2. Sube el contenido del ZIP descomprimido al repositorio. La estructura raíz debe ser: 
    .github/
  workflows/
    deploy.yml       ← el workflow que inyecta los secrets
auth/
  auth.js          ← contiene los placeholders %%GDRIVE_CLIENT_ID%%
  ...
index.html
common/ ...
  3. Activa GitHub Pages:
    • Ve a Settings → Pages
    • Source: "GitHub Actions"
    • (No selecciones una rama — el workflow lo gestiona todo)
ℹ️
Repositorio público vs. privado: Para GitHub Pages gratuito en cuentas personales, el repositorio debe ser público. Las credenciales de Google Cloud no estarán visibles en el código fuente del repositorio (son placeholders), pero sí en el archivo desplegado. Esto es correcto y esperado para el tipo "Desktop app".
4
Añadir los Repository Secrets en GitHub

Los Repository Secrets son variables cifradas que GitHub Actions puede leer durante el build pero que nunca aparecen en los logs ni en el código fuente del repositorio.

  1. En tu repositorio de GitHub, ve a Settings → Secrets and variables → Actions.
  2. Haz clic en "New repository secret".
  3. Crea el primer secret:
    CampoValor
    NameGDRIVE_CLIENT_ID
    SecretEl "ID de cliente" copiado del paso 2c (termina en .apps.googleusercontent.com)
  4. Haz clic en "Add secret".
  5. Crea el segundo secret igual:
    CampoValor
    NameGDRIVE_CLIENT_SECRET
    SecretEl "Secreto de cliente" copiado del paso 2c
✅
Una vez guardados, los secrets nunca se muestran de nuevo — ni siquiera a ti. Solo puedes sobrescribirlos. Esto es por diseño y es correcto.

¿Qué hace el workflow con estos secrets?

El archivo .github/workflows/deploy.yml incluye este paso que sustituye los placeholders en auth/auth.js antes de desplegar:

# Del archivo .github/workflows/deploy.yml
- name: Inject Google Drive credentials
  env:
    GDRIVE_CLIENT_ID:     ${{ secrets.GDRIVE_CLIENT_ID }}
    GDRIVE_CLIENT_SECRET: ${{ secrets.GDRIVE_CLIENT_SECRET }}
  run: |
    sed -i "s|%%GDRIVE_CLIENT_ID%%|${GDRIVE_CLIENT_ID}|g"         auth/auth.js
    sed -i "s|%%GDRIVE_CLIENT_SECRET%%|${GDRIVE_CLIENT_SECRET}|g" auth/auth.js

El código fuente en el repositorio siempre contiene %%GDRIVE_CLIENT_ID%%. Solo el archivo desplegado en GitHub Pages contiene el valor real.

5
Disparar el primer despliegue
  1. Haz un git push a la rama main (o edita cualquier archivo desde la UI de GitHub).
  2. Ve a la pestaña Actions de tu repositorio. Verás el workflow Deploy FinanceApp to GitHub Pages ejecutándose.
  3. Haz clic en él para ver los logs en tiempo real. El paso "Inject Google Drive credentials" debería mostrar: ✅ Credenciales inyectadas correctamente.
  4. Cuando termine (verde ✓), tu app estará disponible en: https://TU_USUARIO.github.io/TU_REPOSITORIO
⚠️
Primera vez: GitHub Pages puede tardar 1-3 minutos en activarse tras el primer despliegue. Refresca la URL si ves un 404.

Lanzar manualmente sin hacer push:

Ve a Actions → Deploy FinanceApp to GitHub Pages → Run workflow. Útil si actualizas los secrets y quieres redesplegar sin tocar el código.

6
Verificar que Google Drive funciona
  1. Abre la URL de tu GitHub Pages en el navegador.
  2. En la pantalla de inicio de FinanceApp, haz clic en "Conectar con Google Drive". Si el botón aparece desactivado con el texto "Google Drive sin configurar", hay un error en los secrets — vuelve al paso 4.
  3. Te redirigirá a Google para autorizar. Inicia sesión con el email que añadiste como usuario de prueba en el paso 2b.
  4. Google mostrará una advertencia de que la app no está verificada. Haz clic en "Continuar" (es tu propia app).
  5. Acepta el permiso "Ver, editar, crear y eliminar únicamente los archivos de la aplicación específica de Google Drive" (scope drive.appdata).
  6. Te redirige de vuelta a FinanceApp. Introduce una contraseña de cifrado y haz clic en "Confirmar". Esta contraseña cifra tus datos antes de subirlos a Drive — Google no puede leerlos.
  7. ¡Listo! Ahora puedes hacer clic en "☁ Drive" en el sidebar para guardar un backup cifrado.
🔐
No olvides la contraseña de cifrado. Si la pierdes, los datos del backup en Drive no se pueden recuperar. Guárdala en un gestor de contraseñas. Los datos locales (localStorage) no están cifrados y siguen accesibles aunque pierdas la contraseña de Drive.

Desarrollo y prueba local

Para probar la app en local con Google Drive funcionando, necesitas sustituir los placeholders manualmente en una copia local. Nunca hagas commit de esta copia con los valores reales.

# Copia de trabajo local (no hacer commit)
cd financeapp
cp auth/auth.js auth/auth.js.backup

# Sustituir placeholders temporalmente
sed -i "s|%%GDRIVE_CLIENT_ID%%|TU_CLIENT_ID.apps.googleusercontent.com|g"  auth/auth.js
sed -i "s|%%GDRIVE_CLIENT_SECRET%%|TU_CLIENT_SECRET|g" auth/auth.js

# Arrancar servidor local
python3 -m http.server 8080
# → http://localhost:8080

# Antes de hacer commit, restaurar los placeholders
cp auth/auth.js.backup auth/auth.js
⚠️
Añade auth/auth.js.backup a tu .gitignore para no cometerlo por accidente. También puedes añadir una regla de pre-commit que verifique que los placeholders están presentes.

Añadir localhost:8080 como URI de redirección: En Google Cloud Console → Credenciales → tu cliente → Añadir http://localhost:8080 como URI de redireccionamiento autorizado.

Preguntas frecuentes
¿Es seguro que el client_secret quede en el JavaScript desplegado?

Para el tipo "Desktop app" (Aplicación de escritorio), sí. Google lo documenta explícitamente: "The client secret is obviously not treated as a secret." El client_secret de una aplicación de escritorio o pública sirve únicamente para identificar a la app ante Google, no para autenticar al usuario. Combinado con PKCE, el flujo es seguro contra ataques de interceptación de código.

Lo que nunca debe ser público es el access_token y el refresh_token del usuario — esos se guardan solo en el localStorage del navegador del usuario y nunca se envían a terceros.

¿Por qué no usar el tipo "Web application" que parece más apropiado para una web?

Técnicamente, en 2026 Google requiere client_secret para todos los tipos de cliente OAuth, incluyendo "Web application". La diferencia es que para "Web application", Google asume que el secret es confidencial (guardado en un servidor backend), lo cual no es el caso de una SPA estática.

El tipo "Desktop app" es idéntico técnicamente pero Google lo declara como cliente público, reconociendo que el secret no puede mantenerse confidencial. Usar "Desktop app" es la práctica documentada y recomendada para SPAs que necesitan acceso a APIs de Google sin backend.

¿Puedo usar la app sin Google Drive?

Sí, perfectamente. Si no configuras los secrets de GitHub (o no añades las credenciales de Google Cloud), la app funciona en modo local: todos los datos se guardan en el localStorage del navegador. El botón de Drive aparece desactivado con una nota explicativa.

Puedes exportar e importar los datos manualmente en formato JSON desde el sidebar.

El workflow falla con "Permission denied" al desplegar a Pages

Ve a Settings → Actions → General → Workflow permissions y asegúrate de que está en "Read and write permissions". Si no, el workflow no tiene permisos para escribir en GitHub Pages.

Alternativamente, verifica que en Settings → Pages el source está configurado como "GitHub Actions" y no como una rama.

Google muestra "redirect_uri_mismatch" al hacer login

La URI de redirección que envía la app no coincide con ninguna de las URIs autorizadas en Google Cloud Console. Comprueba que en la configuración de credenciales (paso 2d) has añadido exactamente la URL correcta.

Casos comunes: URL con / final vs. sin él, URL con www vs. sin él, o falta de http://localhost:8080 para desarrollo local. La URI debe ser una coincidencia exacta.

¿Qué pasa si roto (cambio) las credenciales de Google?

Actualiza los Repository Secrets en GitHub con los nuevos valores y lanza el workflow manualmente (Actions → Run workflow). Esto redespliega la app con las nuevas credenciales. Los usuarios que tenían sesión activa verán que su access_token ha caducado y deberán volver a autenticarse, pero sus datos en Drive permanecen intactos.

¿El backup en Drive es realmente privado?

Sí, en dos niveles: alcance y cifrado.

Alcance (scope): El scope drive.appdata limita el acceso a la carpeta appDataFolder, que es invisible para el usuario desde drive.google.com y para cualquier otra app.

Cifrado: Antes de subir, los datos se cifran con AES-GCM usando una clave derivada de tu contraseña mediante PBKDF2 (200.000 iteraciones). Google solo ve el texto cifrado. Sin tu contraseña, los datos son irrecuperables incluso para Google.

La app desplegada muestra "Google Drive sin configurar" pero sí añadí los secrets

Comprueba el log del workflow en la pestaña Actions. En el paso "Inject Google Drive credentials" debería aparecer "✅ Credenciales inyectadas correctamente.". Si aparece el mensaje de advertencia, los secrets están vacíos.

Verifica que los nombres exactos de los secrets son GDRIVE_CLIENT_ID y GDRIVE_CLIENT_SECRET (sensibles a mayúsculas). Luego lanza el workflow manualmente.
